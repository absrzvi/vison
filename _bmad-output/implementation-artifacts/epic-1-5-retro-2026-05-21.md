## What Went Well

### pydantic-settings v2 kwarg bug caught early
The review of 1-5-3 surfaced that `_make_settings()` in `inference/tests/unit/test_tripwire.py` was passing uppercase kwargs (`INFERENCE_EVENT_STORE_URL`) to `Settings()`, which pydantic-settings v2 silently ignores. The tests had been passing only because the default value happened to match the hardcoded mock URL. Changing `event_store_url` default from `8000` to `8001` exposed the bug. The fix (use field names as kwargs) is now locked in; without the review this silent bug would have persisted into fleet integration tests.

### env_prefix discipline established across all three onboard services
All three Python services now declare `env_prefix` correctly in `SettingsConfigDict` (EVENT_STORE_, RTSP_INGEST_, INFERENCE_). The 1-5-2 review caught the missing prefix on rtsp-ingest; the 1-5-3 review caught the resulting env key mismatch in the compose file. This pattern is now documented in each service's CLAUDE.md and in the smoke test's API_KEY handling.

### Event-store API learning documented
The smoke test debug log captured three non-obvious API facts: (1) the ingest endpoint is `POST /api/v1/events` not `/ingest`; (2) the `EventEnvelope` requires `source` (literal enum) and `severity`, not `source_service`; (3) `GET /api/v1/events?journey_id=...` raises `JOURNEY_NOT_FOUND` unless the journey record exists separately. These facts would otherwise only live in a developer's head — now in the story file and smoke test comments.

### Smoke test CI-hardening in one review pass
The 1-5-4 code review (7 patches) caught every major CI-reliability gap: missing compose v2 pre-flight, no CWD guard, `curl -f` masking 422 bodies, `?limit=1` false-positive on non-fresh volume, missing `cameras.json` stub for `compose config`. All applied in one batch; smoke test still exits 0.

### `service_started` vs `service_healthy` decision documented cleanly
Decision D1 in 1-5-3 (whether inference should depend on rtsp-ingest `service_healthy`) was resolved with a deliberate "relax to `service_started` for dev; revert to `service_healthy` in production" note baked into the compose file itself. Future engineers operating on SYS2 have the context inline.

## What Didn't Go Well

### Story ACs written before API was verified — rotted immediately
AC4 and AC5 in the 1-5-4 story spec referenced `/api/v1/ingest` and `PASSENGER_BOARDED` — neither exists. The ACs were written from memory of the story intent, not from reading the actual event-store routes. This caused three wasted implementation iterations (wrong endpoint, wrong event type, wrong GET filter). The spec text was not corrected before merge; the debug log documents what actually happened, but the ACs remain stale.

### No stub `cameras.json` in repo root
The `docker-compose.onboard.yml` bind-mounts `./cameras.json` for rtsp-ingest and inference, but there is no `cameras.json` in the repo. `docker compose config` validates the full file — on a fresh clone, CI fails at config validation before a single container starts. The smoke test works around this by creating a stub at runtime, but the real fix (commit a minimal `cameras.json` stub to the repo root, or split the compose file into per-service files) was deferred.

### Smoke test `journey_id` hardcodes a date
`"SMOKE-TEST_001_20260521"` in the POST body will silently rot if the `EventEnvelope` ever gains temporal date validation. The fix (`$(date -u +%Y%m%d)`) is trivial but was deferred. Should have been applied during implementation.

### `HEALTH_TIMEOUT=30` is too tight for cold-image build runs
On a runner with no Docker layer cache, the `--build` step for event-store can exceed 30s, leaving no time for the container to start before the health poll times out. This manifested during development (required re-running the test after the image was cached). The timeout should be separated from the build step.

## Patterns to Carry Forward

- **Read the actual routes before writing ACs** — never write API paths from memory; grep `routes/` first
- **`env_prefix` checklist** — every new Python service needs `env_prefix` in `SettingsConfigDict` and a matching CLAUDE.md env-var table; verify compose env keys match the prefix
- **Smoke test pre-flight is not optional** — `docker compose version`, CWD guard, stub file creation for bind-mount sources; these prevent CI failures that have nothing to do with the code under test
- **`curl -f` is wrong for status-code inspection** — drop `-f` any time you need to read the HTTP status or body for diagnostics; use `-s -o body.txt -w "%{http_code}"` instead
