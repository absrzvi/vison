## The Headline Pattern

**5 of 6 stories carried a BLOCKER into code review, and every one was the same species: green tests that did not exercise the real production path (test/prod divergence).**

| Blocker | How it hid | How it was proven closed |
|---|---|---|
| 10-1 token-type mismatch 422s cloud-sync | re-validation boundary untested | contract test extended to WAGON_EXIT/ENTRY — the test that 422'd now passes |
| 10-4 `seconds_to_departure` plumbing dead in prod | synthetic-kwargs unit tests masked nested-pis vs flat-key | R2 drove **real** `update_pis` → real `/context` → value=599 stamped through live producer→fusion→KPI |
| 10-5 integration tests violate FK (orphan `escalation_id`) | unit tests never hit DB; integration "deferred to CI" | **ran `pytest -m integration` on real Postgres — 6/6 + 92/92 full suite pass** |
| 10-6 missing `id` for CC matching | — | patch + RED-verified test |
| 10-1 fusi-1 500 on crafted payload | handler caught only HTTPError | RED-verified test, `except Exception` guard |

**Closing discipline graded:** every found blocker is now closed with a test that fails on old code / passes on new, run on the real path (for 10-4 and 10-5, the live wire was literally driven). The review's *closing* discipline is strong.

**The real exposure:** the blockers were caught by **discretionary** adversarial-wire review, not an always-on gate. Had the review not run, 5 blockers ship. Confidence must rest on a reliable net, not on per-story diligence.

## What Went Well

- **Adversarial-wire review caught all 5 blockers before ship.** The [adversarial-review memory] practice — replay the REAL producer wire, not synthetic kwargs — worked exactly as designed. 10-5's completeness critic found the FK blocker all 4 find-layers missed.
- **Epic-4 A1 lesson applied successfully.** Epic-4 phase-2 retro: "ADR updates must be a story checkbox, not retro guidance." This epic, 10-6 shipped **ADR-21 in the same commit** via the codified ADR-FRESHNESS create-story rule. The continuity loop closed.
- **create-story payload-audit caught divergence before code.** 10-4 (4 epic↔shipped divergences) and 10-5 (5) were surfaced as story Decisions *before* implementation — the cheapest place. This is the A3 lesson confirmed again.
- **The epic's trust theme held in the work.** 10-1 confidence chips, 10-5 two-rates-not-one, 10-3 SOP — all about not lying to the operator with a single fake number. The 10-5 review even caught us re-introducing the exact ambiguity (rounded `5.0%` vs raw 5% gate) the epic exists to kill.
- **Leaner BMAD flow sped development up** (Project Lead observation) without losing the load-bearing ceremony (payload-audit).
- **A misdiagnosed "blocker" was corrected, not accepted.** 10-5's "browser-verify blocked" was a self-inflicted misread (pushState+reload raced React's mount); re-checked properly and discharged AC7 live.

## What Didn't Go Well

- **Pattern 1 — test/prod divergence is systemic, not incidental.** 5 of 6 stories. Root cause: unit tests were allowed to stand in for integration tests, and integration tests didn't run locally ("deferred to CI" was the cover). The review is currently doing the job of *both* the first and last line of defence.
- **Pattern 2 — review rigor was inconsistent and operator-dependent.** 10-5 got a 14-agent workflow; 10-6 a single round. Depth tracked how dangerous a story *felt*, not a rule. Low-risk stories (10-3 docs, 10-5 additive-read-only) got heavyweight treatment; the budget wasn't aimed where blockers live.
- **Pattern 3 — round-2s were the biggest time sink.** 10-1 went 3 rounds, 10-4 and 10-5 went 2 each. Each round is a full re-review. The integration gate (A1) would have made 10-4 and 10-5 one round each.
- **A docs story shipped a false safety guarantee.** 10-3 §3.4 "never silently dropped" contradicted `truncate_old_journeys(retain=3)`. Caught in review and bounded — but a written safety claim is the kind that gets read aloud after an incident.

---

## Key Insights for Future Stories

- **The fix is to make the review need to catch less.** The blockers are a pre-review-gate problem, not a review-quality problem. Move integration-test execution upstream of `review` and 3 of 5 blockers die at the dev desk.
- **Both of the Project Lead's concerns (more confidence, more speed) resolve to the same two actions.** The integration gate makes review *more* trustworthy AND eliminates the round-2 rework that was the biggest time cost. Risk-tiering frees the heavyweight review to fire only where blockers live. No safety/speed trade — one move buys both.
- **Epic 11 raises the stakes on this exact discipline.** Auth/Identity swaps real JWT into every endpoint currently behind the `X-API-Key`/`X-Admin-Key` seam. An auth regression that only shows against a real token store is precisely the test/prod-divergence species this epic was burned by — the hard integration gate matters *more* there.
