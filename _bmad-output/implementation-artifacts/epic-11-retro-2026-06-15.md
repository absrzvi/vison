## What Went Well

1. **Epic-10's process fixes worked — measurably.** Epic 10's thesis was "5 of 6 stories carried the same blocker species (green tests not exercising the real path), caught only by discretionary review." Epic 11 wired A1/A2/A3 into the harness *before* the first story. Outcome: **zero blockers into review, every story one round.** The A1 integration-before-review gate is the single highest-leverage change made across these two epics.
2. **The adversarial review caught real bugs even with clean dev.** Zero blockers ≠ nothing to find. S4 caught a body-actor **spoof vector** (audit actor from a request field, not the token). S5 caught a genuine **fail-open** — an admin could set a "valid" floor of 0.0 and silently disable the degraded banner fleet-wide, contradicting the story's own invariant. Both passed all green tests; the FULL tier (A2) is why they didn't ship.
3. **The ADR-23 verification seam held across four consecutive stories.** S1 built `_verify_token` as the OIDC-swap seam; S2's liveness check, S4's role swap, and S5's admin PATCH all composed on it without touching the seam. The design paid for itself.
4. **create-story PAYLOAD-AUDIT / ADR-FRESHNESS earned their keep.** S3 caught that 0011 was already taken (user_audit numbering, stale in the epic); S5 caught two divergences (migration must be 0012; there is no per-class confidence gate — per-class values are display-only). Both surfaced as story Decisions before any code.
5. **Surgical scope discipline across five auth stories.** S3 didn't rebuild preferences (already done by S1); S5 didn't touch the operator preference or the read-only chip client. No scope creep.
6. **Every story shipped a minimal functional, browser-verified screen** (login, Users, Profile, Alert Classes, Configuration) so the backend critical path never waited on design — and so Freya's upcoming WDS pass works on real screens, not mockups.

## What Was Harder (no blame — systems & tooling)

1. **Docker flakiness nearly broke the A1 gate.** On 11-4 the integration gate was blocked ~26 min because Docker Desktop wouldn't start (engine service needs elevation). The story was correctly HELD in review rather than flipped on unit coverage — the gate did its job — but it cost wall-clock and needed Abbas to start Docker. Tooling fragility, not process.
2. **A recurring same-species defect S3→S5: the "valid-but-wrong-at-a-boundary value."** S3's UUID-vs-str response-model 500; S5's float-0.0 fail-open; S5's float32 display artifact. Different surfaces, same shape — a value that's structurally/type valid but semantically wrong at a boundary. Party-mode (below) refined this: the three examples actually split across THREE gates (create-story, dev-story type-discipline, browser-verify), so one rule can't own all three.
3. **Browser-verify is heavy and Docker-coupled.** Each story stood up disposable PG + uvicorn + Vite. Faithful but ceremonious — and S5's float-precision bug was findable *only* in the browser, vindicating the mandate while underscoring its cost.

## Party-Mode Decision — a 6th create-story rule (Winston's point)

**Motion:** add a create-story rule forcing enumeration of semantically-valid-but-wrong-at-a-boundary values.

**Debate (Winston / Mary / Freya / Amelia):**
- Winston: the existing five rules catch *contract* divergence; none ask *which in-range values are wrong*. Three-for-three this epic.
- Mary: the three examples are not one family — S5's 0.0 is boundary-semantics; S3's UUID-500 is a *type-seam* issue; the float32 artifact is *storage/display precision*. One broad rule would mean nothing.
- Freya: the float32 bug was browser-only (exact in DB + API) → belongs to verify-in-browser, not create-story. The three examples split across three gates.
- Amelia: existing rules are cheap because *mechanical* ("compare X to Y"). A "imagine the bad values" rule is *generative* → gets skipped or balloons. It must be a checklist, not a meditation.

**Consensus (Abbas's call: TIGHT scope):** YES to a 6th rule, scoped narrowly to **values that drive a gate/threshold/comparison**, phrased as three mechanical questions — NOT a general "enumerate bad values" rule (that's what adversarial review is for). The UUID-seam case is a dev-story type-discipline concern, not create-story; the float-display case is already owned by the browser-verify mandate.
