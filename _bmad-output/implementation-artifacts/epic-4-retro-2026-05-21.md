## What Went Well

### Every story shipped against the strict quality gate
≥90 % coverage on every container, `mypy --strict`, `ruff` zero violations. The bar held across nine stories without exception. 4-4 hit 99.29 % after rework; 4-6 hit 98.76 %.

### Code review caught real defects in every story
Examples that would have shipped without the review pass:
- **4-4 (meta-review):** dual-budget bypass, RTSP URL leak path, bbox pixel-vs-normalised confusion, in-flight guard, shutdown TOCTOU. 17 meta-items + 19 review patches before APPROVED.
- **4-6:** missing `asyncio.Lock` race in suppression state machine; ramp_deployed edge-trigger debounce missing — would have re-fired on every push; `Settings.journey_id` placeholder removed in favour of skip-on-None.
- **4-7:** replay-vs-live race in WS fan-out (closed via register-first + per-subscriber dedupe set); `?after=<unknown>` silently returning page 1 (now 400 INVALID_CURSOR); `EVENT_STORE_API_KEY=""` "looks-configured-but-unreachable" footgun normalised to None.
- **4-8 (round-2):** confidence never propagated from `accepted` dicts to TripwireHandler; `_last_side` `dict` → `OrderedDict` for true LRU; orphan timer cancel-before-emit ordering fixed.

### Party mode as a real decision instrument
Three stories went to party-mode roundtable rather than the dev agent making silent judgement calls: 4-4 D1–D3 (Winston/Amelia/Freya), 4-8 D1+D2 (Winston/Amelia/John), and 4-9 D1–D5 (Winston/Amelia/Saga/Freya). Every roundtable produced changes that the original spec author hadn't anticipated — 4-9's D5 rename (`LEDGER_DRIFT_ALERT` → `LEDGER_DRIFT_OBSERVATION`) being the most consequential.

### A1 (full UPDATE-file reads) was applied consistently
Every Phase 1 story has explicit "Existing Files Being Modified" sections in Dev Notes with current-state / what-changes / what-must-be-preserved bullets. The pattern Amelia recommended in the E3 retro stuck and almost certainly prevented regressions across 4-4 → 4-5 → 4-6 → 4-8 (all four touched overlapping inference + fusion paths).

### A3 (async stale-closure pattern) prevented bugs in 4-4 and 4-8
The `loop_holder` pattern from 4-4 was reused verbatim in 4-8's tripwire orphan timer. Round-2 review on 4-8 (P10) caught the cancel-before-emit ordering — directly the stale-closure family of issue the E3 retro flagged.

### `pipeline.py` exclusion pattern scaled across containers
Established in 4-3 (`rtsp-ingest`), reused identically in 4-4 (`inference`). Hardware-dependent GStreamer code stays in a single excluded file; everything else is unit-testable. Removed the "we can't test this until we have a Hailo" excuse from the entire pipeline tier.

### `respx.mock` as a universal HTTP test pattern
Adopted across inference, fusion, event-store, cloud-sync, and the new tripwire tests. No real HTTP traffic in any unit suite. Test runs are fast and deterministic.

### Spec-vs-reality gaps caught in `create-story`, not in code review
4-9 and 4-10 both had divergent payload shapes between `epics.md` and the shipped `shared/` contracts. Both were caught during `bmad-create-story` (this session) rather than after dev agents tried to implement to the wrong spec. Story-creation as a spec-audit step is paying off.

## What Didn't Go Well

### Pattern 1: 4-4 was a meta-review event, not a single-pass story
**Numbers:** 19 code-review patches + 17 meta-review patches + decisions D1–D3 settled via party-mode mid-story. That's 36 patch items before APPROVED — by far the highest in the epic.

**Root cause:** 4-4 was the architectural pivot from "Python tracker (BYTETracker) + Python callback" to "TAPPAS-native `hailotracker` + thin callback." The pivot happened DURING the story (commit `d87bb71` — "E4-S4 pivot to TAPPAS-native architecture"), not before. The story was being re-specced while it was being implemented.

**Cost:** Re-work cost was high. The meta-review caught real bugs (M1 dual budget, M4 asyncio.Lock, M11 shutdown TOCTOU, M13 journey_id) that the first review missed because the architectural surface kept shifting.

**Lesson:** When a story is mid-flight and an architectural pivot becomes necessary, **abort the story, re-spec it, restart**. Continuing to patch incrementally produced more total work than a clean restart would have.

### Pattern 2: 4-5 round-1 review left tests failing
**Numbers:** Code-review (2026-05-20) gate evidence: `pytest 117 PASS / 2 FAIL`. Story was marked `review → in-progress` until the failures were addressed. 20 review/follow-up items resolved before final APPROVED.

**Root cause:** The story was sent to review with a known red bar. The "100 % pass before review" discipline from Amelia's role description was violated.

**Lesson:** Never send a story to review with red tests. The review loop should be reading code, not chasing test failures.

### Pattern 3: Spec/reality drift in `epics.md` payload definitions
**Examples:**
- 4-9 epic spec: `LedgerDriftAlertPayload = {expected_total, actual_total, delta, coach_breakdown, reconciliation_applied}` — aggregate.
  Shipped `shared/`: `{car_id, camera_count, ledger_count, delta, threshold}` — per-coach.
- 4-10 epic spec: `CoachComfortIndexPayload = {car_id, reserved_seats, occupied_seats, standing_count, comfort_score, service_tier}`.
  Shipped `shared/`: `{car_id, comfort_score, occupancy_pct, temperature_c, noise_db}`.

**Root cause:** `epics.md` was written upfront in BMAD Phase 3, before E1-S2 (event envelope + payloads) shipped. When the payloads were actually implemented in `shared/`, the team made better decisions, but `epics.md` was never updated to reflect them.

**Cost:** Caught in create-story for 4-9/4-10 (good), but only because the create-story workflow now reads `shared/` before trusting `epics.md`. Earlier stories in the epic may have absorbed similar drift silently.

**Lesson:** `epics.md` is a snapshot, not a source of truth. The actual `shared/` contracts and ADRs win. Create-story should explicitly compare epic-spec payload definitions against the shipped `shared/` models and flag divergence.

### Pattern 4: ADR-17 (closed-ledger) was specified before reality
ADR-17 §line 451 says "the `local-fusion-engine` maintains a `coach_ledger` table in SQLite" — but as of the end of Phase 1, `fusion` is **stateless in-memory**, with no SQLite anywhere. 4-9 will introduce the first SQLite into fusion. The ADR predicted state that doesn't exist yet.

ADR-17 also describes an aggregate `LEDGER_DRIFT_ALERT` event that was renamed to `LEDGER_DRIFT_OBSERVATION` in the 4-9 party mode, with shape entirely unlike the ADR's description.

**Lesson:** ADRs need a revision date / "current as of" stamp. When an ADR is contradicted by shipped reality, update the ADR before the next story that touches it (4-9 should ship with an ADR-17 update).

### Pattern 5: `fusion` is the first cross-cutting consumer and didn't have an ingest path
When 4-9 (and now 4-10) needed `OCCUPANCY_UPDATE`/`WAGON_EXIT`/`WAGON_ENTRY` events, it turned out `fusion` had no path to consume them — only `/context` from vlan-pollers and `/candidates/*` from inference for door_obstruction + slip_fall. The "event-store WS fan-out" (4-7) was built for landside consumers, not for onboard fan-in.

**Root cause:** No architectural pass on "who needs to consume events emitted by inference?" — only on who needs to consume events on the **landside**.

**Cost:** 4-9 had to invent the ingest pattern (D1-A: extend `/candidates/*` for `wagon_*` and `occupancy_update`). 4-10 piggybacks on it. Both stories ended up larger than they would have if fusion had had a generic event-ingest hook from the start.

**Lesson:** Before the next epic, audit which on-edge containers need to be event consumers (not just producers) and decide whether they all go via `/candidates/*` POST-from-producer or via a shared event-store WS subscription pattern.

### Pattern 6: `4-5-code-review-2026-05-20-followup.md` is a structural anti-pattern
This file exists as a separate artifact for the round-2 review patches. It's a story file that isn't a story — it's review feedback in story shape.

**Lesson:** Round-2 review feedback should live inside the original story file as a "Senior Developer Review (AI) — Round 2" section (the pattern 4-8 uses). One story → one file → many review rounds.

## Summary

Epic 4 Phase 1 was a success in delivery (9 of 9 in-flight stories shipped to a strict quality bar) and a strain test for the review loop. 152 review patches across 9 stories is a lot of catches — meaning the dev agents produced a lot of defects, AND the review process worked. The two are inseparable.

The single biggest learning is that **the spec → code path drifts**: `epics.md` payloads diverged from shipped `shared/` contracts, ADR-17 diverged from shipped reality, and 4-4's architectural pivot happened during the story rather than before it. The discipline to catch and fix drift early (in create-story rather than in code review, in ADR updates rather than after-the-fact) is the next leverage point.

Phase 1 also validated the party-mode tooling as a real decision instrument. 4-4, 4-8, and 4-9 all reached better resolutions in roundtable than the dev agent would have alone. The pattern is worth preserving and using deliberately when a story has more than two open decisions touching different concerns.

4-9 and 4-10 will run with these lessons in hand. A focused Phase 2 retro can close out the loop when they ship.
