## What Didn't Go Well

### Pattern 1: High review patch counts in early stories (E3-S1, E3-S2)
**Root cause:** Dev agent entered stories without full context on the files being modified. When updating a router, it needs to see the full router tree. When touching a migration, it needs the full migration history.

**Examples:**
- E3-S1: Python built-in `range` shadowed by a local variable — subtle test failure
- E3-S2: Five root causes — router composition, missing unique constraint, URL.revokeObjectURL race, activeRangeRef race

### Pattern 2: CSS token errors
- E3-S5: `--obb-sev-high` used — token doesn't exist; correct token is `--obb-sev-critical`
- E3-S7: Same category of error

**Root cause:** The canonical CSS token list in `src/index.css` was never surfaced in story Dev Notes. Dev agent had to discover tokens by reading the CSS file — and sometimes guessed wrong.

### Pattern 3: React hooks edge cases discovered during implementation (not prevented)
- E3-S3: `react-hooks/set-state-in-effect` suppression needed
- E3-S6: `react-refresh/only-export-components` triggered by exporting constant from context file  
- E3-S7: Stale closure in async `useCallback` — needed `useRef` mirror pattern to read state synchronously

**Root cause:** These are known React patterns that should be documented in project context so the dev agent anticipates them rather than discovering them mid-implementation.

## Summary

Epic 3 was a success by the most important measure: we shipped. The review process caught real security and correctness issues. The main process improvement opportunity is upstream — giving the dev agent better context going in (full file reads, CSS token list) to reduce the review patch load.

Epic 4 is a fundamentally different technical domain. The learnings from E3 (especially A1 + A3) apply directly: async Python callbacks have the same stale-state risks as React hooks, and reading existing code before modifying it is always worth the extra step.
