# CLAUDE.md — cloud-backend

FastAPI service (Python 3.11+). Landside REST API + SSE push for the Control Centre Dashboard. Connects to PostgreSQL (asyncpg + SQLAlchemy async). Depends on `oebb-shared` for event schemas.

## Stack

- FastAPI + uvicorn
- SQLAlchemy 2.x async + asyncpg driver
- Alembic for migrations
- Pydantic v2 + pydantic-settings for config
- structlog for structured logging
- testcontainers[postgres] for integration tests (real Postgres, no mocks)

## Commands

```bash
cd cloud-backend
pip install -e ".[dev]"
python -m pytest                        # all tests
python -m pytest -m unit               # fast, no DB
python -m pytest -m integration        # needs Docker for testcontainers
python -m ruff check src/
python -m mypy src/
uvicorn cloud_backend.main:app --reload  # dev server on :8000
```

Coverage threshold: 80%.

## Module Layout

```
src/cloud_backend/
  routes/     — one file per resource group (fleet, alerts, escalations, health)
  api/        — request/response Pydantic models
  services/   — business logic invoked by route handlers
  config/     — pydantic-settings Settings; all env vars here
  database.py — async engine + session factory
  main.py     — FastAPI app factory; mounts routers
```

## File Conventions

- Routes in `routes/` contain route handlers only — no business logic
- Pydantic models in `api/` — separate from SQLAlchemy ORM models
- New env var → add to the `config/` Settings class first, then `.env.example`
- Migration: `alembic revision --autogenerate -m "description"` then review before applying

## What NOT to Touch

- `migrations/versions/` — generated by Alembic; edit only the generated file, never hand-craft SQL here
- Do not add synchronous DB calls in async route handlers (blocks the event loop)
- Never hardcode secrets — everything through `pydantic-settings` / env vars

## Key Patterns

SSE push: the `/events` endpoint streams from an asyncio queue populated by the ingest worker. Do not poll the DB on each SSE tick — push from the queue.

Auth (E11-S1 / ADR-23): self-contained JWT. Protected routers attach `Security(get_current_user)` (Authorization: Bearer) — auth is per-router, there is no central middleware. `get_current_user` decodes/validates the token (via the single `_verify_token` core in `api/auth.py`, which reads issuer/algorithm/key from `Settings` only) and yields a `CurrentUser{user_id, username, role}`. Role-gate a route with `Security(require_role("admin"))` (a dependency factory) — not by checking the role inside the body. The SSE stream (`/api/v1/alerts/stream`) uses `get_current_user_from_query` (`?token=`) because EventSource can't set a header. `login` and the infra probes (`/health/live`, `/health/ready`) stay unauthenticated. `JWT_SECRET` empty fails closed. The legacy `X-API-Key` (`require_api_key`) is retired for routing but stays live for the machine-to-machine `POST /api/v1/events` ingest per ADR-23 (a per-producer service identity is Phase-2, tracked E11S3-D4).

Kill-switch (E11-S4): the alert-class kill-switch router (`routes/admin_alert_classes.py`, `POST .../{alert_code}/{disable|enable}` + the GET list) is now `require_role("admin")` (JWT) — the old shared `X-Admin-Key`/`CC_ADMIN_KEY` is RETIRED (the `cc_admin_key` Setting was removed; the old curl path 401s). The two POST handlers take a per-route `current: CurrentUser = Security(require_role("admin"))` so the audit actor is `current_user.username`, never a client-supplied body field — `AdminActionBody` was deleted, the POST body is empty. The fan-out filter (`services/fanout_filter.py`) and the three-way audit (alert_class_state row + ALERT_CLASS_DISABLED/REENABLED event envelope + structured log) are unchanged by the auth swap.

Liveness (E11-S2): both extractors run an `is_active` lookup (`assert_user_active`) AFTER `_verify_token`, so deactivating a user invalidates their existing token mid-session, not just their next login. This is authorization, layered in the extractor — `_verify_token` stays pure (the ADR-23 OIDC-swap seam). Never fold the `is_active` SELECT into `_verify_token`. `bcrypt_rounds` is config-driven (prod default 12); a Settings validator rejects `< 10` unless `app_env == "test"` (so the cheap test cost can't leak into prod). Admin user-management (`routes/users.py`, `require_role("admin")`) audits every mutation to the local `user_audit` table (Alembic 0010) + a structured log, in the same transaction as the mutation — NOT via the event envelope (user-account actions are landside-local). The last active admin cannot be deactivated/demoted (atomic `SELECT … FOR UPDATE` guard → 409).

Preferences (E11-S3): `operator_preferences.operator_id` is a UUID FK to `users.user_id` (Alembic 0011). It was re-keyed from the old shared-API-key TEXT string — the defunct rows were dropped (drop/seed, Decision D1: no honest 1-key→N-user mapping existed), not migrated. `routes/preferences.py` keys the GET/PATCH by `current_user.user_id` (token-only identity; any body `operator_id` is ignored). The raw-SQL `RETURNING operator_id` yields a UUID object → coerce to `str` for the `str`-typed response model (the model stays `str`, the column is `uuid`). `unattended_threshold_min` is localStorage-only (no server column) and is NOT on the Profile screen.

Confidence thresholds (E11-S5): the per-class confidence thresholds + the `degraded_banner_floor` are persisted in the `confidence_thresholds` KV table (Alembic 0012; `config_key` PK = `per_class:<code>` or `degraded_banner_floor`, `value` double, range CHECK) and admin-mutable via `PATCH /api/v1/config/confidence-thresholds` (`require_role("admin")`, actor = `current_user.username`; the GET stays operator-readable for the UnifiedFeed ConfidenceChip). Consumers read through `config/confidence_thresholds.py::threshold_store` (a `ThresholdStore` cache mirroring `fanout_filter` — TTL + lock + double-check + generation + `invalidate()`), NOT the module constants — a module-level capture would never see an edit. The module's `DEFAULT_CONFIDENCE_THRESHOLDS` / `DEGRADED_BANNER_FLOOR` remain the FAIL-SAFE fallback: any missing/NULL/non-finite/out-of-range stored row falls back to the hardcoded default per key, so a malformed store can never disable the gate (fails SAFE, never open). The only live backend gate is `health.py`'s `ai_quality_degraded` (rolling-1h mean < floor); per-class thresholds are display-only (ConfidenceChip) — there is no per-class suppression gate.

Error shape: all 4xx/5xx responses use `{"detail": "..."}` — never return raw exception messages to clients.

## Review Failure Scenarios

Every story touching this service must verify these scenarios before sign-off:

- **SSE worker restart:** stream must recover without duplicate event delivery to connected clients
- **Migration safety:** every Alembic migration must be safe under concurrent reads — no full-table locks on the `events` or `alerts` tables
- **JWT edge cases:** expired token, tampered signature, and missing role claim must all return 401/403 — never a 500
- **SSE queue backpressure:** a slow SSE client must not block the ingest worker — drop or bound the per-client queue

## Untrusted Input Boundary

MQTT payloads arriving from cloud-sync originate from onboard sensors — treat as untrusted at the landside boundary. Validate against the shared event schema before persisting to PostgreSQL. Do not pass raw MQTT payload fields into SQL queries or SSE frames without schema validation.
